# 任务二算法流程图

## 一、算法流程说明

任务二旨在将自然语言问题自动映射为结构化查询、中文回答与图表结果，并进一步导出为符合赛题要求的标准提交文件。由于题目同时包含多轮上下文承接、结构化 SQL 生成、结果质量控制和图表输出等要求，其算法设计不能停留在单步问答层面，而应构建包含解析、判定、修复、渲染和导出的多阶段处理流程。

当前实现以 LangGraph 状态机为主线，将任务二拆分为若干职责明确的节点，并通过共享状态维护当前轮次、槽位解析结果、SQL 历史、查询结果与图表路径等中间信息，从而使整个求解过程具备可追踪、可修复和可复现的特征。

## 二、总体算法流程

图 1 展示了任务二从问题输入到结果导出的总体算法流程。

```mermaid
flowchart TD
    A["读取附件4问题集"] --> B["逐题初始化状态"]
    B --> C["逐轮读取子问题"]
    C --> D["累计上下文并解析槽位<br/>公司、期间、指标、图表意图"]
    D --> E["判断是否缺失关键信息"]
    E -->|是| F["生成澄清回答"]
    E -->|否| G["构建 query_plan"]
    G --> H["生成 SQL"]
    H --> I["执行 SQL 并校验结果"]
    I -->|失败且未超过重试上限| H
    I -->|成功| J["生成 chart_plan"]
    J --> K["构建 chart_spec 并渲染图片"]
    K --> L["生成正式中文回答"]
    F --> M["写入当前轮 Q/A/image"]
    L --> M
    M --> N["轮次结果聚合"]
    N --> O["导出单题 answer_json 与 graph_format_text"]
    O --> P["批量汇总导出结果文件与调试工件"]
```

## 三、问题解析与澄清流程

任务二中的多轮问题具有明显的上下文承接特征，因此问题解析过程需要同时考虑当前轮文本与历史轮次信息。图 2 展示了该部分算法流程。

```mermaid
flowchart TD
    A["输入当前轮问题"] --> B["拼接累计问题文本"]
    B --> C["识别公司简称与股票代码"]
    C --> D["识别报告期表达"]
    D --> E["识别指标、TopN、阈值与图表类型"]
    E --> F["结合上一轮上下文做承接补全"]
    F --> G["形成 parsed_slots"]
    G --> H["检查公司、期间、指标是否完整"]
    H -->|缺失| I["生成 missing_slots"]
    I --> J["输出澄清型回答"]
    H -->|完整| K["进入查询规划阶段"]
```

图 2 任务二问题解析与澄清流程

## 四、查询生成与自动修复流程

自然语言问题即使已经完成槽位解析，也不一定能够一次生成正确 SQL。为此，系统采用“生成 - 执行 - 反馈 - 修复”的闭环算法。图 3 给出了该流程。

```mermaid
flowchart TD
    A["输入 query_plan"] --> B["LLM 生成 SQL"]
    B --> C["SQL 安全检查<br/>仅允许 SELECT/WITH 且必须查询 financials_view"]
    C --> D["执行 SQL"]
    D --> E["结果后处理<br/>过滤关键指标为空记录"]
    E --> F["结果有效性校验"]
    F -->|通过| G["输出 result_rows / result_preview"]
    F -->|不通过| H["记录 sql_error"]
    H --> I["判断 sql_attempts 是否小于 3"]
    I -->|是| J["将 SQL 与报错信息回送模型修复"]
    J --> B
    I -->|否| K["结束本轮查询并进入 warning / error 分支"]
```

图 3 任务二查询生成与自动修复流程

## 五、图表规划与渲染流程

为提高图表输出的可控性与复现性，系统没有直接让模型输出绘图结果，而是采用“规划 - 规格化 - 渲染”的三层算法。图 4 展示了该部分流程。

```mermaid
flowchart TD
    A["输入查询结果 dataframe"] --> B["规则生成默认 chart_plan"]
    B --> C["LLM 结构化修正字段与排序方式"]
    C --> D["形成 chart_plan"]
    D --> E["转换为 chart_spec"]
    E --> F["保存 .spec.json 工件"]
    F --> G["按 spec 进行渲染"]
    G --> H["生成 ./result/题号_序号.jpg"]
    H --> I["写入当前轮 A.image"]
```

图 4 任务二图表规划与渲染流程

## 六、批量运行与结果导出流程

单题执行结束后，系统还需要将全部题目结果汇总为正式提交文件，并同步保留调试信息。图 5 展示了批量导出流程。

## 七、算法步骤概括

从实现角度看，任务二算法可概括为以下步骤：

1. 读取问题汇总、公司信息和任务一数据库，并构建统一查询视图 `financials_view`。
2. 对每道题逐轮初始化状态，并拼接累计问题文本。
3. 解析公司、报告期、指标、图表类型、TopN 和阈值等槽位信息。
4. 结合上一轮上下文判断信息是否完整，并在必要时生成澄清回答。
5. 对信息充分的问题构建结构化 `query_plan` 并生成 SQL。
6. 对 SQL 执行结果进行安全校验、有效性判定和自动修复重试。
7. 对有效结果生成正式回答，并在需要时采用结构化枚举兜底。
8. 对需要绘图的问题生成 `chart_plan`、`chart_spec` 并输出图片文件。
9. 将每轮 `Q/A/image` 写入会话结构，形成最终 `answer_json`。
10. 汇总全部题目结果，导出提交主表、调试文件和中间工件。

## 八、算法流程的意义

该算法流程的核心价值在于将任务二从一次性生成答案的黑箱过程，转化为可拆解、可回放、可调优的显式求解链路。与单步自然语言转 SQL 的方法相比，本文算法在多轮承接、错误修复和图表规范输出方面具有更高的稳定性，也更符合数学建模竞赛中“结果可解释、过程可复核”的要求。
