# 任务二论文正文小节初稿

## 3.1 问题分析

任务二要求系统针对题目给出的自然语言问题，自动完成财务数据查询、中文回答生成和图表输出，并按规定格式导出到 `result_2.xlsx`。与一般单轮数据库问答不同，本题既包含单公司指标查询，也包含行业筛选、排名统计、趋势分析和多轮追问等复杂场景；同时，输出结果不仅需要文字答案，还要求在必要时生成图像文件并将其路径写入回答结构。因此，任务二本质上并非单纯的“自然语言转 SQL”问题，而是一个集多轮意图理解、结构化查询、结果校验和可视化表达于一体的综合工作流问题。

从数据依赖关系看，任务二以前一任务形成的结构化财务数据库为基础，不再直接面向原始 PDF 文档。这意味着其核心难点已由“信息抽取”转变为“问题理解与查询生成”。然而，赛题中的问题表达具有较强灵活性，同一财务指标可能以“营收”“营业总收入”“收入增长率”“营收增长率”等不同方式出现，报告期也可能以“2025年第三季度”“去年”“近三年”等多种形式描述，这为问题解析和口径统一带来了较大挑战。

另一方面，多轮问答中的上下文依赖也是任务二的重要难点。例如，首轮问题可能先给出企业集合，后续轮次再以“这些公司”“其中哪家”等代词承接前文。若系统将每一轮都视为孤立问题，则很难正确恢复查询对象和语义范围。因此，任务二必须显式保留会话状态，使后续轮次能够继承前序结果中的公司集合、指标语境和查询上下文。

此外，即使模型能够生成语法正确的 SQL，也并不意味着结果一定满足业务需求。查询结果仍可能存在记录过少、趋势时间点不足、比例尺度异常或筛选范围不合理等问题。若缺少执行后的结果有效性校验与自动修复机制，系统容易输出形式正确但含义错误的答案。基于上述分析，本文认为任务二应采用面向状态机的整体框架，将问题解析、查询规划、SQL 修复、图表生成与结果导出纳入统一流程，以保证系统的稳定性、可解释性与可扩展性。

## 3.2 方法设计

### 3.2.1 总体设计思路

针对任务二的多轮问数与结果交付需求，本文构建了“统一数据视图 - 多轮槽位解析 - 查询计划生成 - SQL 生成与自动修复 - 结果校验 - 图表规划与渲染 - 回答生成 - 提交导出”的处理框架。该框架以 LangGraph 状态图为核心，将任务二拆分为若干职责明确的节点，并通过共享状态维护当前轮次、累计问题文本、解析槽位、查询计划、SQL 历史、查询结果、图表路径和轮次回答等中间信息。

与线性脚本流程相比，这种状态机设计更适合处理多轮问答中的条件分支与重试逻辑。一方面，它能够将“是否需要澄清”“是否需要修复 SQL”“是否需要生成图表”等判断显式化；另一方面，也能为每一题保留完整状态快照，从而便于后续调试与论文分析。

### 3.2.2 统一查询视图构建

任务二的数据基础来自任务一生成的四张业务表，包括利润表、核心业绩指标表、资产负债表和现金流量表。若直接让模型面向多表生成 SQL，不仅会增加 Prompt 负担，也容易出现联表逻辑复杂、字段口径不统一和查询表达不稳定等问题。为此，本文在运行时将四张业务表按 `stock_code + report_period + report_year` 拼接为统一宽表 `financials_view`，并将其作为任务二的唯一查询入口。

在统一视图构建过程中，系统还会对部分字段进行融合与派生。例如，当同一指标在不同业务表中具有候选来源时，优先按稳定来源进行补齐；当研发费用与营业总收入同时存在时，可进一步计算研发费用占比；当存货与营业成本同时存在时，可推导存货周转率。该设计既降低了查询生成复杂度，也提高了结果表达的一致性。

### 3.2.3 多轮问题解析与上下文承接

针对赛题中的多轮问答场景，系统首先对当前轮问题与历史轮次文本进行累计拼接，再通过规则解析器完成公司、股票代码、报告期、指标、图表类型、TopN 和阈值条件的抽取。在公司识别上，系统结合公司简称、股票代码和少量常见别名实现统一匹配；在报告期识别上，系统支持年度、半年度、季度及前三季度等规范表达，并对“去年”“近三年”等常见描述进行基础映射；在指标识别上，则利用关键词映射实现自然语言表达与标准财务指标之间的对应。

为处理“这些公司”“上述企业”“其中哪家”等承接性表述，系统会将上一轮查询结果中的公司集合和部分结果记录保存在共享状态中。当当前轮未显式给出公司名、但命中明显承接模式时，系统优先继承上一轮上下文，从而恢复完整查询对象。该机制显著增强了多轮问答场景下的问题理解能力。

### 3.2.4 查询计划生成与 SQL 自动修复

在得到基础槽位后，系统首先构建结构化 `query_plan`，再据此调用大语言模型生成 SQL。`query_plan` 不仅保留公司、期间和指标等核心信息，也记录题型、图表意图、TopN 和阈值条件等补充约束，从而为 SQL 生成提供更稳定的输入结构。为保证执行安全，系统限制 SQL 只能为只读语句，且必须访问 `financials_view`，禁止任何破坏性数据库操作。

考虑到模型首次生成的 SQL 仍可能存在错误，本文进一步设计了自动修复回路。当 SQL 在语法校验、执行阶段或结果有效性检查中失败时，系统会将上一版 SQL 与错误信息一并回送模型，请其基于已有结果进行修正，最多重试 3 次。该机制可处理的典型问题包括：排名题返回记录不足、趋势题时间点过少、报告期格式异常、结果数值全零以及比例字段尺度明显异常等。相比一次性生成 SQL 的方案，这种闭环修复机制更适合复杂业务问数任务。

### 3.2.5 结果校验与回答生成

SQL 查询成功后，系统并不直接将结果送入回答模块，而是先执行后处理与质量检查。系统会根据问题中涉及的核心指标筛除无效记录，并检查排名题的样本规模、趋势题的时间覆盖以及比例类字段的数值尺度。只有通过这些检查后，结果才进入回答生成阶段。

在回答组织方面，系统区分两类情形。若当前轮缺少关键槽位，则生成澄清型回答，引导补充公司、期间或指标；若结果已经有效返回，则生成正式中文回答。对于“列出”“展示”“分别是多少”等结构化需求较强的问题，系统还会检测回答内容是否完整覆盖公司和关键字段；若覆盖不足，则采用确定性枚举输出作为兜底。该设计兼顾了自然语言表达的流畅性与结果信息的完整性。

### 3.2.6 图表规划与 Spec 驱动渲染

部分问题明确要求绘制趋势图、柱状图或其他统计图，因此任务二还需完成图表输出。本文未直接让模型决定最终绘图细节，而是采用“默认规则选图 - LLM 结构化修正 - Spec 驱动渲染”的三层方案。首先，系统根据问题语义和结果表结构推断默认图表类型，例如趋势问题优先使用折线图，排名或对比问题优先使用柱状图，占比问题优先使用饼图。随后，模型对图表的横轴字段、纵轴字段、分组字段、排序方式和标题进行结构化修正。

在此基础上，系统继续将 `chart_plan` 转换为标准化 `chart_spec`。该 spec 明确记录数据来源、字段编码、排序方式、布局参数和输出文件名，使渲染器仅承担确定性执行职责。通过这一设计，图表生成由“模型直接绘图”转变为“模型规划、程序执行”，从而提高了图表输出的一致性、可控性和可复现性。

### 3.2.7 批量运行与结果导出

在单题闭环基础上，系统进一步支持单题调试、按题号批量运行、随机抽样和全量运行。批量运行器会统计每题状态，并同步输出 `result_2.xlsx`、图片目录、调试 CSV、运行汇总 JSON、每题 debug 状态和图表 spec 工件。这样不仅能够满足赛题最终提交要求，也为 Prompt 调优、误差分析和论文撰写提供了充分的中间证据。

## 3.3 方法特点

与传统的一次性问答方式相比，本文方法具有以下几个特点。第一，系统采用显式状态机组织多轮问数流程，使澄清分支、SQL 修复分支和图表分支能够统一纳入可追踪工作流。第二，系统通过统一宽表 `financials_view` 降低了多表财务查询的复杂度，提高了结构化查询生成的一致性。第三，系统在 SQL 执行后增加了结果有效性校验和自动修复机制，从而有效缓解了“语法正确但业务错误”的问题。第四，图表模块采用 spec 驱动渲染，使可视化结果具备更好的结构化表达与复现能力。第五，系统在输出正式结果的同时保留调试状态与中间工件，为后续优化提供了可靠依据。

## 3.4 误差分析与局限性

尽管当前任务二已经形成较完整的求解闭环，但仍存在若干局限。首先，问题解析阶段仍包含一定规则依赖，对于表达高度省略、别名不常见或口语化较强的问题，槽位抽取仍可能不充分。其次，SQL 自动修复虽然能够缓解部分执行失败问题，但其修复仍依赖当前错误反馈，若问题本身存在复杂业务歧义，模型仍可能生成语法正确但口径偏差的查询结果。

再次，当前结果校验主要聚焦于样本规模、时间覆盖和数值尺度等显性问题，对于更深层的业务语义偏差，如统计口径不一致、排序维度误判或图表表达形式不优等问题，仍需进一步引入更细粒度的规则或人工复核。最后，当前系统主要依赖任务一形成的结构化数据库，当问题超出数据库字段范围、需要引入解释性文本证据时，任务二仍缺少更强的外部检索与证据融合能力。

因此，后续优化方向不在于推翻现有框架，而在于围绕当前状态机继续细化节点能力，包括增强别名解析与时间表达识别、补充业务一致性校验、扩展图表模板库，并逐步引入面向任务三的文本检索与证据融合模块。

## 3.5 与后续任务的衔接

任务二在整体系统中起到承上启下的作用。其上游依赖任务一形成的结构化财务数据库，其下游则直接面向任务三所需的增强分析与多源证据融合。当前架构已经将问题解析、结构化查询、图表生成和回答组织拆分为独立节点，因此后续只需在此基础上继续增加检索、证据重排序、SQL 与文本证据融合以及自检节点，即可平滑扩展为更完整的任务三工作流。

因此，任务二的价值不仅在于完成当前赛题要求的智能问数任务，更在于建立了一套可复用、可扩展、可调优的中间工作流框架，为后续多源增强分析提供了坚实基础。
