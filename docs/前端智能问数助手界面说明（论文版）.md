# 上市公司财报“智能问数”助手前端界面设计

## 1. 界面建设目标

为提升模型系统的可解释性、交互性与现场展示效果，本文在三项任务求解框架之外构建了一个独立的 Web 问答界面，命名为上市公司财报“智能问数”助手。该界面不改变原有任务一、任务二、任务三的核心求解代码，而是在任务三的问答、检索、SQL 查询、证据融合和图表生成能力之上增加交互层封装，使评委或用户能够以自然语言方式提出问题，并直观获得回答文本、生成图表与研报引用依据。

界面设计遵循“问题输入—任务调用—答案呈现—证据复核”的交互闭环。用户在输入框中提交问题后，系统将问题传入后端问答服务，由后端复用任务三中已经构建的图式流程完成意图识别、财务数据库查询、研报证据检索、跨源信息融合和最终回答生成。前端随后将模型返回的结构化结果拆分展示为回答内容、生成图像和引用依据，从而使模型输出不仅可读，而且可追溯、可复核。

## 2. 总体架构

前端界面采用轻量化浏览器页面实现，后端采用本地 HTTP 服务承接请求。整体结构可概括为：

\[
Q \rightarrow \mathcal{P}(Q) \rightarrow \mathcal{D}(Q) \oplus \mathcal{R}(Q) \rightarrow \mathcal{F}(Q) \rightarrow A
\]

其中，\(Q\) 表示用户输入问题；\(\mathcal{P}(Q)\) 表示问题解析与任务规划；\(\mathcal{D}(Q)\) 表示结构化财务数据库查询结果；\(\mathcal{R}(Q)\) 表示研报文本与图表证据检索结果；\(\oplus\) 表示结构化数据与非结构化证据的融合；\(\mathcal{F}(Q)\) 表示答案组织、图表渲染与自检过程；\(A\) 为最终展示给用户的结构化回答。

该设计使前端并不直接参与模型推理，而是作为任务三推理链路的可视化交互入口。其优势在于：一方面保持原有算法流程的稳定性，避免为了界面展示而改动核心求解逻辑；另一方面使系统具备更强的人机交互能力，便于展示问答系统在真实使用场景中的连续追问、证据引用和图表生成效果。

## 3. 问答交互与连续追问

界面主体由问题输入框、回答框、图片框和引用信息组成。用户可以直接输入单轮问题，也可以在已有回答之后继续输入追问。对于追问型问题，系统在同一会话中保留历史问题序列：

\[
H_t=\{q_1,q_2,\ldots,q_t\}
\]

其中，\(H_t\) 表示第 \(t\) 轮问答时的上下文集合。后端将历史问题序列组装为多轮任务记录，使第 \(t\) 轮问题能够在前序语义基础上继续解析。例如用户先询问某企业近几年收入变化，再追问“上升原因是什么”，系统可将后者解释为对前一企业和前一指标的延续分析，而不是孤立问题。

回答框采用对话式结构展示每一轮问题与回答，并在每轮答案下方嵌入引用依据。该方式避免了多轮回答混杂在同一长文本中造成的信息混乱，也便于用户逐轮检查模型对问题意图的理解是否一致。

## 4. 第三问题库菜单与题组自动执行

为适应比赛现场复核和评委快速选题的需求，界面右上方设置了第三问题库菜单。鼠标悬停后，系统展示任务三全部题目，并按题号、题型和首问内容组织为可点击条目。菜单支持题号、题型和关键词筛选，便于在较多题目中快速定位目标问题。

对于多轮题，系统在选中题目后自动识别其子问题序列：

\[
\mathcal{Q}_i=\{q_{i,1},q_{i,2},\ldots,q_{i,m}\}
\]

其中，\(\mathcal{Q}_i\) 表示第 \(i\) 道题的完整题组，\(m\) 为该题包含的子问题数量。界面提供两种执行方式：第一种是手动逐问执行，用户点击“第 1 问”“第 2 问”等按钮后逐次发送；第二种是“顺序执行本题”，系统自动按照原题顺序依次提交全部子问题。自动执行过程等价于：

\[
A_{i,k}=G(q_{i,k}\mid q_{i,1},q_{i,2},\ldots,q_{i,k-1}),\quad k=1,2,\ldots,m
\]

其中，\(G(\cdot)\) 表示后端问答生成函数。该机制能够展示系统处理连续追问题的能力，同时减少人工复制题目的操作成本，使比赛演示过程更稳定。

## 5. 生成状态与取消机制

由于任务三问答过程涉及数据库查询、研报检索、大模型调用和图表渲染，单次回答可能需要一定时间。为避免用户误以为页面卡顿，界面在生成过程中显示实时计时和阶段状态，包括问题解析、查询规划、证据检索、信息融合和回答复核等步骤。其状态序列可表示为：

\[
S=\{s_1,s_2,s_3,s_4,s_5\}
\]

其中，\(s_1\) 为问题解析，\(s_2\) 为数据库查询规划，\(s_3\) 为研报证据检索，\(s_4\) 为信息融合，\(s_5\) 为回答复核。前端根据等待时间动态更新当前提示，使用户能够感知系统正在推进。

同时，界面提供“取消生成”按钮。当用户触发取消时，前端向后端发送取消标记，后端在各关键流程节点之间检查该标记，并在当前计算步骤结束后停止后续流程。由于大模型请求或图表渲染等外部调用通常无法在函数内部立即中断，因此该取消机制采用协作式取消策略：

\[
C_t =
\begin{cases}
1, & \text{用户发出取消请求}\\
0, & \text{继续执行}
\end{cases}
\]

在每个主要节点执行前后检查 \(C_t\)。若 \(C_t=1\)，则终止剩余节点，返回取消状态。该设计兼顾了系统稳定性与用户控制感，避免强制中断导致中间状态损坏。

## 6. 回答质量提示与证据呈现

为了增强输出结果的可复核性，界面在每轮回答上方加入质量提示标签。标签依据回答内容和返回结构自动生成，主要包括回答完整度、研报引用数量、生成图表数量和是否包含数值信息等。设回答文本为 \(a\)，研报引用集合为 \(R\)，生成图表集合为 \(I\)，则质量提示可抽象为：

\[
\phi(a,R,I)=\{\mathbb{I}(|a|\geq \tau), |R|, |I|, \mathbb{I}(a\ \text{contains numeric values})\}
\]

其中，\(|a|\) 表示回答文本长度，\(\tau\) 为完整度提示阈值，\(|R|\) 为研报引用数量，\(|I|\) 为生成图表数量，\(\mathbb{I}(\cdot)\) 为指示函数。该提示并不替代正式评分，而是为用户提供快速复核线索：若回答缺少研报引用，用户可进一步检查问题是否需要文本证据；若问题要求可视化而图表数量为零，则可提示用户重新组织问题或检查图表链路。

引用区嵌入每轮回答下方，展示研报标题和引用图表。对于存在研报原文片段的结果，界面采用折叠方式展示引用原文，既保证页面简洁，也允许用户在需要时展开核验。该设计体现了检索增强生成系统的证据约束思想，即答案不应只是语言模型的直接生成，而应尽可能绑定到可追溯的信息来源。

## 7. 图表展示与结果导出

当后端生成图表时，前端会在图片框中展示一张或多张图像，并支持点击查看大图。该功能用于展示任务三中“结构化财务数据—可视化图表”的生成链路，使趋势类、对比类、排名类问题的答案更直观。

此外，界面提供复制最新回答、导出 Markdown 和导出 JSON 的功能。Markdown 结果适合人工阅读和论文材料整理，JSON 结果保留了模型输出的结构化字段，便于后续复核和二次分析。结果导出结构保持与后端输出一致，包括问题、回答正文、图片列表和引用证据。

## 8. 临时输出与关闭清理

在交互式问答过程中，系统需要为图表渲染、查询缓存和前端图片展示生成临时文件。为了避免演示过程产生的临时结果混入正式提交结果，前端服务将本次交互产生的中间文件统一放置在独立的临时输出空间中。关闭网页、心跳断开或终端中止服务时，系统自动触发清理流程，删除本次前端交互生成的缓存、图片和中间结果。

该清理机制可表示为：

\[
\Omega_{\text{web}}=\Omega_{\text{cache}}\cup\Omega_{\text{chart}}\cup\Omega_{\text{artifact}}
\]

其中，\(\Omega_{\text{web}}\) 为前端交互产生的临时输出集合，\(\Omega_{\text{cache}}\) 为运行缓存，\(\Omega_{\text{chart}}\) 为图表文件，\(\Omega_{\text{artifact}}\) 为查询和检索中间结果。服务关闭时执行：

\[
\operatorname{Clean}(\Omega_{\text{web}})=\varnothing
\]

同时，清理范围被限制在前端交互专用输出空间内，不删除任务一、任务二、任务三的正式结果和知识库索引。该设计既保证演示过程可以正常生成图表和缓存，又避免本地目录在多次演示后积累无关文件。

## 9. 前端界面的作用

该 Web 界面是对三项任务求解系统的交互式封装，其核心作用不在于替代模型算法，而在于提升系统可用性和可展示性。通过题库菜单、连续追问、自动执行、取消控制、质量提示、图表展示和引用复核等设计，系统能够从“批处理式结果生成”扩展为“面向用户的智能问答助手”。这使比赛成果更接近真实业务中的财报问答系统：用户提出问题，系统调用结构化数据库和非结构化研报证据，最终返回带有可视化结果和引用来源的答案。

从系统工程角度看，该界面遵循低侵入原则，仅在原有任务三推理链路外增加适配层，不改变任务一、任务二、任务三的核心代码。因此，前端展示能力和后端求解能力保持了解耦关系，既便于比赛演示，也便于后续扩展为企业财报分析、行业研报问答和投资研究辅助工具。
