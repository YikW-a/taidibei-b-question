# 任务三全量建库与小样本回答验证推进顺序

## 1. 当前总体策略

任务三建议按“两段式”推进：

1. 先把**数据准备层**做稳
2. 再做**小样本回答验证**

不要一开始就直接跑全量回答，因为任务三当前仍处在：

- 知识库质量验证期
- retrieval 策略验证期
- references 格式收口期

更稳的方式是：

- 先确认知识库质量
- 再确认检索效果
- 最后再确认回答链

---

## 2. 推荐推进顺序

### 阶段 A：重建并确认知识库

适用场景：

- 改了 chunk 规则
- 改了 metadata 标准化
- 改了 embedding 模型
- 改了 vector store 结构

建议执行：

```bash
rm -rf outputs/task3_langgraph
python run_task3_index.py \
  --embedding-batch-size 64 \
  --embedding-batch-pause-seconds 1
```

先确认：

- `report_chunks.json` 正常生成
- `report_metadata_lookup.json` 正常生成
- `index.faiss` 正常生成
- `index_meta.json` 中：
  - `index_type = faiss_flat_ip`
  - `embedding_model = BAAI/bge-m3`

这一阶段的目标不是全量建完，而是先确认：

- PDF 抽取没问题
- chunk 结构没问题
- embedding 能打通
- FAISS 能落盘

---

### 阶段 B：小样本向量建库稳定性验证

在阶段 A 没问题后，继续使用当前统一的建库参数跑到全量完成。

建议命令：

```bash
rm -rf outputs/task3_langgraph
python run_task3_index.py \
  --embedding-batch-size 64 \
  --embedding-batch-pause-seconds 1
```

观察：

- 是否出现 `413 / 429`
- 是否能稳定推进 `completed_chunk_count`
- `chunk_quality_samples.json` 是否比前一版更干净
- `index_meta.chunk_count` 是否持续增长

这一阶段目标是验证：

- 批量 embedding 稳定
- 当前 chunk 清洗策略有效
- 当前批大小和停顿时间可接受

---

### 阶段 C：全量建库

当前项目约定已经统一为：

```bash
rm -rf outputs/task3_langgraph
python run_task3_index.py \
  --embedding-batch-size 64 \
  --embedding-batch-pause-seconds 1
```

然后等待建库推进，直到：

- `remaining_chunk_count = 0`
- `index_meta.chunk_count` 接近 `report_chunks.json` 的总 chunk 数

这一阶段目标是：

- 获得真正可用的全量 vector 知识库

---

### 阶段 D：检索冒烟

在全量知识库建好后，不要立刻跑回答链，先做 retrieval smoke test。

建议命令：

```bash
python run_task3_index.py \
  --retrieval-smoke-question "请分析云南白药盈利改善的驱动因素" \
  --retrieval-mode hybrid
```

重点看：

- 命中的 `paper_path`
- chunk 文本是否相关
- `metadata` / `vector` / `hybrid` 的差异
- `references` 基础字段是否可回查

这一阶段目标是：

- 先验证“证据找得准不准”

---

### 阶段 E：3 到 5 题小样本回答冒烟

检索冒烟没问题后，再进入回答链。

建议先选 3 到 5 题，覆盖：

- `rag_only`
- `sql_only`
- `hybrid_sql_rag`
- 多轮题

例如之前已经验证过的小样本：

```bash
/opt/anaconda3/envs/taidibei/bin/python run_task3_langgraph.py --question-ids B2002,B2003,B2005,B2008
```

重点看：

- `result_3.xlsx`
- `artifacts/debug/*.json`
- SQL 是否合理
- retrieval 证据是否合理
- `references` 是否接近附件 7

---

### 阶段 F：收口与调优

只有在前面都稳定后，再做这些优化：

- retrieval plan 调优
- rerank 调优
- answer prompt 调优
- self_check 调优
- references 格式对齐
- 任务三图表链

---

## 3. 当前最推荐的实际执行顺序

现在最合适的顺序是：

1. 按统一命令重建最新知识库
2. 检查 `chunk_quality_samples.json`
3. 跑到全量向量索引 ready
4. retrieval smoke
5. 3 到 5 题回答冒烟
6. 小样本调优
7. 再扩大到更多题

---

## 4. 当前 chunk 质量优化方向

当前 chunk 已从“按页滑窗”升级为：

- 标题/段落优先
- 图表/表格标题优先
- 超长正文再滑窗

并且新增了这些优化：

- 过滤页码、声明、来源等噪声行
- 过滤封面评级、分析师、公司简介、免责声明等整段噪声
- 对超长密集文本按句号/分号进一步切开
- 合并过短 chunk，避免碎片化过强
- 默认最多读取每篇 PDF `100` 页；若 PDF 不足 `100` 页，则自动读到最后一页

也就是说，现在的 chunk 不再是简单按页硬切。

## 6. 现在的建议

如果当前目标是稳步推进任务三，不建议直接切去大批量回答。

更稳的方式是：

1. 先把全量知识库建完
2. 再做 retrieval smoke
3. 再做小样本回答验证

这样排查问题会清楚得多：

- 是 chunk 问题
- 是 embedding / FAISS 问题
- 是 retrieval 问题
- 还是回答链问题
