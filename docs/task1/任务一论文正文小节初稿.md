# 任务一论文正文小节初稿

## 3.1 问题分析

任务一要求从上市公司财务报告 PDF 中提取关键财务数据，并按照给定数据库结构完成入库。这一任务的本质是面向复杂半结构化文档的结构化信息抽取问题，其主要难点体现在以下几个方面。

首先，财务报告的版式差异显著。不同交易所、不同公司乃至不同年份的财报，在表头命名、单位表达、分页方式和续表形式上均存在较大差异。例如，同一类指标可能以“归属于上市公司股东的扣除非经常性损益的净利润”“归属于本公司股东的扣除非经常性损益的净利润”等多种方式出现，给字段统一带来较大挑战。其次，财报中的目标报表并非始终处于固定页面，且经常出现跨页延续、标题拆行、数值换行等现象，导致单一表格抽取工具难以稳定适配。再次，任务要求并不仅限于文本读取，还要求完成字段标准化、单位统一、数据校验和数据库入库，因此必须在抽取环节之后进一步设计结果融合与质量控制机制。

从任务依赖关系看，任务一是后续任务二和任务三的重要基础。任务二中的自然语言问数模块依赖于任务一输出的结构化财务数据库；任务三中的数据核验、归因分析和多源增强问答同样需要将财务数据作为核心证据来源。因此，任务一的目标不仅是“抽取得到结果”，更是构建一个结果稳定、结构统一、可追溯、可扩展的数据底座。

基于上述分析，本文认为任务一不宜采用单步骤、单工具的抽取方式，而应构建一条由“预处理、抽取、转换、融合、校验、评估”组成的闭环处理流程，以兼顾抽取准确率、结果一致性与后续应用可用性。

## 3.2 方法设计

### 3.2.1 总体设计思路

针对财务报告 PDF 结构复杂、版式差异大和字段口径不统一等问题，本文构建了“数据预处理 - 多策略抽取 - 标准化转换 - 跨表融合 - 数据校验 - 数据入库 - 质量评估”的任务一处理框架。该框架以四张目标表为中心，即核心业绩指标表、资产负债表、现金流量表和利润表，分别完成表定位、字段抽取和结果入库，并在流水线末端输出质量评估结果和人工核验材料。

### 3.2.2 数据预处理

在数据预处理阶段，本文首先对上交所和深交所财报目录进行统一规范。对于上交所数据，主要根据文件名解析股票代码、报告日期和随机标识，生成结构化清单；对于深交所数据，则进一步识别“摘要”“全文”“更正后”“更新后”等版本后缀，并依据“更正版优先”原则完成版本归并。经过预处理后，系统能够得到统一格式的 `manifest.csv/json` 文件，从而为后续批量抽取提供稳定输入。

### 3.2.3 多策略报表抽取

在抽取阶段，本文采用多策略结合的方法。首先利用 `PyMuPDF` 对 PDF 页面进行扫描，并依据“合并资产负债表”“合并利润表”“合并现金流量表”“主要会计数据和财务指标”等关键词定位目标报表的起始页。随后，针对跨页报表场景，使用 `pdfplumber` 进行跨页表格抽取，以增强对财务报表续页结构的处理能力。

考虑到部分报表在 PDF 中并不能以标准表格对象形式稳定提取，本文进一步引入 `fitz.combined_text` 文本兜底方案，对拆行标签和续行数值进行重构；在单页结构化表格场景下，则使用 `PyMuPDF.find_tables()`、`pdfplumber.extract_tables()` 和 `camelot.read_pdf()` 形成 fallback 链路。通过该多策略抽取设计，系统能够在不同版式条件下保持较高的适应性。

### 3.2.4 字段映射与单位统一

抽取结果获得后，本文构建字段别名映射词典，将财报中的中文项目名统一映射至数据库标准字段。例如，“归属于上市公司股东的净利润”“归属于本公司股东的净利润”等被统一映射到同一目标字段。与此同时，对金额字段执行单位统一，将 `元`、`万元` 和 `亿元` 转换为统一尺度；对比例类字段则统一处理 `%`、百分点、括号负数等特殊格式。通过字段映射与单位统一，可以显著降低版式差异对结构化结果的一致性影响。

### 3.2.5 候选结果融合与跨表补值

由于同一份财报可能从多个页面得到多条候选记录，本文按照 `stock_code + report_period + report_year + table_name` 对候选结果进行分组，并优先保留字段更完整、来源更稳定的候选项。除此之外，本文进一步利用跨表关系执行补值与派生计算。例如，可由资产总计和负债合计推导所有者权益合计，可由利润表和资产负债表结果补充核心业绩指标表中的毛利率、净利率、每股净资产和扣非净资产收益率等字段。该机制能够显著提升目标字段覆盖率，尤其对核心业绩指标表效果较为明显。

### 3.2.6 数据校验与入库

在形成最终结构化记录后，本文对数据执行多层规则校验，并在入库前完成一致性处理。第一层为基础完整性校验，主要包括必填字段检查和业务主键重复检查，用于保证同一公司、同一报告期、同一报表类型的记录具备可入库的关键元数据，避免出现主键缺失或重复写入问题。第二层为数值合理性校验，重点针对同比、环比、比例类字段执行异常值检测，以识别明显超出正常口径范围的可疑结果。第三层为勾稽一致性处理，利用报表间的逻辑关系对结果进行补全和修正，例如由资产总计与负债合计推导所有者权益合计，由经营、投资和筹资活动现金流净额合成净现金流，并进一步推导资产负债率和现金流占比等指标。

在数据库落地策略上，系统默认使用 SQLite 进行本地持久化，以方便调试和中间结果保留；同时支持通过传入 MySQL 连接串将结果直接写入 MySQL 数据库，以满足赛题数据库应用场景的要求。为满足题目对输出表结构一致性的要求，系统在导出阶段严格按照附件 3 的字段定义对四张业务表进行补列与重排，并将最终结果单独输出到 `outputs/task1/final_tables/` 目录。入库完成后，本文又设计了独立的会计勾稽校验脚本，对数据库内部逻辑进行二次验证，并输出通过率、失败率和失败样例，从而使“入库前自动校验”和“入库后二次质检”形成闭环。

### 3.2.7 质量评估与结果交付

为了保证任务一结果的可解释性与可复核性，本文在主流程末端设计了质量评估模块。该模块能够输出字段覆盖率统计、缺失字段频次统计、缺失原因分析、人工抽样核验模板以及论文结果表和文字稿。该设计使得任务一不仅能够给出数据库结果，还能够给出结果质量的量化描述与人工复核依据。

## 3.3 实验结果

为验证本文方法的有效性，本文在正式数据上进行了全量实验。经过清洗与版本归并后，系统共有效处理 1235 份财务报告，并最终形成 3290 条主记录。其中，资产负债表、现金流量表、核心业绩指标表和利润表的记录数分别为 820、823、824 和 823 条。对应的平均字段覆盖率分别达到 0.9327、0.8785、0.8017 和 0.9229。

从结果分布来看，资产负债表和利润表已达到较高覆盖水平，现金流量表也表现出较好的稳定性，说明本文提出的多策略抽取与跨表补值机制能够有效应对标准财务报表中的多数版式变化。与开发初期相比，跨页表格抽取、文本兜底抽取和候选结果融合显著提升了表尾项目、续页项目以及拆行项目的识别效果。

从校验结果看，当前全量结果已无 `error`，仅剩 17 条 `warning`，表明系统已经消除了主键缺失、关键元数据遗漏和整表落空等结构性问题。剩余告警主要集中在少量同比或比例字段的异常值，属于长尾版式差异带来的局部问题，而非流程性失效。核心业绩指标表的平均覆盖率相对较低，但进一步分析表明，其主要缺失字段为 `net_profit_qoq_growth` 和 `operating_revenue_qoq_growth`，且缺失原因多被判定为“源文档可能未披露”，因此这类缺失具有一定客观性，并不能简单视为抽取失败。

在数据库落地之后，本文进一步执行了基于会计勾稽关系的二次复核。结果显示，共构建 11 条勾稽校验规则，形成 8021 个可判定样本，其中通过 7592 个、失败 429 个，整体通过率达到 94.65%。从规则分布看，资产负债平衡、资产负债率一致性、现金流占比一致性、净利率一致性和每股指标一致性均取得较高通过率；相对失败率较高的规则主要为现金流合计勾稽和 ROE 跨表一致性，这说明当前系统在标准主表项目上的内部逻辑已基本稳定，而长尾问题更多集中在少量跨表口径差异和现金流子项抽取偏差上。

整体而言，实验结果表明本文提出的任务一处理框架能够较为稳定地完成从财务报告 PDF 到结构化数据库的转换任务，并已具备为后续问数与增强分析任务提供数据支持的能力。

## 3.4 误差分析

尽管当前方法已经取得较高覆盖率，但仍存在少量误差来源，主要体现在以下几个方面。

第一，少数长尾字段仍受财报版式差异影响。例如部分公司在“合同负债”“短期借款”等字段上的布局与常见版式不同，或者字段被拆分到多行、多列，导致现有规则未能完全覆盖。第二，部分可派生字段依赖于上游输入的完整性。例如“每股经营现金流”“每股净资产”等指标需要同时具备现金流、净利润、股本或权益等基础字段，一旦其中任一字段缺失，则派生链条会受到影响。第三，部分字段的缺失并不来自抽取算法，而是源于原文披露限制。对于年度报告而言，季度环比类指标往往并非标准披露内容，因此这类字段缺失具有一定客观性。第四，即使基础记录已经入库，仍可能存在少量跨表口径不完全一致的情况，因此本文额外设计了会计勾稽校验脚本，对总资产平衡、现金流合计和关键财务比率进行数据库级复核，以进一步识别潜在异常记录。

从当前结果看，任务一系统的主要误差已从早期的“整体抽取不足”转变为“少量长尾字段与披露差异问题”。结合数据库级勾稽校验结果可知，当前最值得重点复核的方向并非资产负债表和利润表的主体结构，而是现金流量表子项的口径一致性以及核心业绩指标表与利润表、资产负债表之间的个别跨表比率差异。这表明后续优化不应再集中于整体流程重构，而应聚焦于个别长尾字段规则补充、少量特殊版式适配以及人工抽样复核机制的完善。这样的优化方向更符合竞赛场景下的时间投入与性能收益平衡。

## 3.5 与后续任务的衔接

任务一完成后形成的四张结构化业务表，为任务二的自然语言问数提供了可直接调用的数据源，也为任务三中的数据校验、归因分析和多源增强问答提供了财务证据基础。因此，任务一的贡献不仅体现在“抽取结果本身”，更体现在建立了一套可被后续任务复用的数据底座。换言之，任务一的结构化程度和数据质量，直接决定了后续系统整体性能的上限。
